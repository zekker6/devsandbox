# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased](https://github.com/zekker6/devsandbox/compare/v0.17.3...HEAD)

### Breaking Changes

- **bwrap proxy mode now requires `iproute2` and `nft` or `iptables` on the host, and its sandboxes are IPv4-only.** The egress lockdown described under *Changed* is applied with the host's `ip` and firewall binaries inside the sandbox network namespace, so a proxy-mode launch (`--proxy`) on a host missing `ip`, missing both `nft` and `iptables`, or without the `nf_tables`/`ip_tables` and `nf_conntrack` kernel modules the rules need - now **aborts** naming what is missing, instead of starting with egress open. The refusal comes from a pre-flight that applies the real rule set in a throwaway namespace before the sandbox is started, so the launch fails with `proxy mode needs an enforceable egress lockdown` rather than an opaque status mid-launch; `devsandbox doctor` reports the same check as the `proxy: firewall` row. There is no opt-out: a lockdown that silently enforced less than promised is exactly the failure the change exists to prevent. Proxy-mode bwrap sandboxes are also now IPv4-only (`pasta -4`), matching what the krun and docker backends already do - the ruleset is IPv4, and leaving IPv6 configured in the namespace would leave a second family for which nothing is filtered. Non-proxy launches are unaffected in every respect: same arguments, no lockdown, no firewall dependency, IPv6 unchanged.
- **bwrap proxy mode now also requires a `pasta` that supports `--map-host-loopback`.** That option is what maps the proxy gateway (`10.0.2.2`) to the host, and the egress lockdown's only permitted destination is that gateway - so on an older `pasta` the sandbox would get a deny-by-default firewall whose one exception points at an address that maps to nothing, and every connection, the proxy's included, would hang until it timed out with nothing said about why. The launch is now refused up front naming the missing option. Only proxy mode is affected; a non-proxy launch on such a `pasta` behaves exactly as before. The embedded `pasta` always supports it, so this is reachable only with `use_embedded = false` or a failed extraction.
- **The container image no longer bakes in `gh` (GitHub CLI) or `neovim` by default; the image is slimmer.** These are now off unless the image is rebuilt with `--build-arg INSTALL_GH=true` / `--build-arg INSTALL_NVIM=true`, or added in a derived `FROM ghcr.io/zekker6/devsandbox` Dockerfile. Their host configs are still mounted into the sandbox as before; only the baked binaries changed. `node`, `claude`, `codex`, `opencode`, `mise`, `git`, the shells, and `docker` CLI remain baked. The Dockerfile also gained `SANDBOX_HOME`/`SANDBOX_UID`/`SANDBOX_GID`/`NODE_VERSION` build args (alongside `INSTALL_GH`/`INSTALL_NVIM`) so the sandbox account and baked node version are each defined once instead of being repeated throughout the file.

### Added

- **New `rtk` tool, so the [rtk](https://github.com/rtk-ai/rtk) CLI proxy keeps its configuration and history inside the sandbox.** rtk is normally installed as a Claude Code hook, so it runs on nearly every command an agent issues, but nothing mounted its directories - your global filters and `config.toml` were invisible in the sandbox, and the tracking database behind `rtk gain` was recreated empty on every launch. `~/.config/rtk` (`config.toml`, `filters.toml`, `filters/*.toml`) is now mounted with the default tmpoverlay, so your filters apply while `rtk config --create` inside the sandbox cannot rewrite them, and `~/.local/share/rtk` (`history.db`, `trusted_filters.json`, `tee/`, `hook-audit.log`, telemetry markers) gets a persistent overlay, so savings recorded in the sandbox accumulate across runs of that sandbox and the host's database is never modified. Both are optional and skipped when absent. rtk resolves these through XDG, which devsandbox points back at the host home path, so the host paths bound are exactly the paths rtk resolves in-sandbox. A project's own `.rtk/filters.toml` is in the project directory and was already writable; `rtk trust` marks it trusted inside the sandbox only. See [Tools: rtk CLI Proxy](docs/tools.md#rtk-cli-proxy).
- **herdr can now capture and restore a sandboxed agent's native session.** herdr records an agent's own session identity (Claude's `--resume` ID, Pi's transcript path) so it can resume that agent after a server restart, but the integration's reports were denied by the proxy's deny-by-default allowlist, so a session started inside devsandbox was never recorded. A new `agent_reporting` capability permits exactly three methods - `pane.report_agent_session`, `pane.report_agent` and `pane.release_agent` - and nothing else; `pane.clear_agent_authority` stays denied, because it revokes another party's claim on a pane. It grants no pane reads, no input injection and no host execution. The capability activates only for a direct `devsandbox claude|pi|codex` launch inside a herdr pane, because that is the only case where devsandbox knows both anchors every report is checked against: the pane herdr created for this process (`HERDR_PANE_ID`) and the agent named on the command line. Neither comes from the sandbox, so a report for another pane or claiming another agent is denied. Starting a shell with `devsandbox` and typing `claude` inside it does **not** enable it. Reports may carry a session ID (at most 128 bytes of `[A-Za-z0-9._-]`, because herdr shell-quotes it and types it into a host pane shell), a transcript path (absolute, at most 4096 bytes, no control characters, restricted to `[A-Za-z0-9._-]` plus the path separator, no `..` component, and lexically inside the launched agent's own session directory as the sandbox sees it - the charset matters because herdr persists a path-kind session ref exactly as it persists an ID and types either back into a host pane shell on restore, and Pi reports only a path), and for lifecycle reports one of the three states a shipped integration sends (`idle`, `working`, `blocked`) plus a 512-byte control-character-free status message. The session-directory bound is host-derived from each agent's own bindings, so it cannot drift from where sessions are actually persisted, and a report arriving with no bound configured is denied rather than waved through. Confinement is deliberately lexical: the reported path names a directory inside the sandbox's overlay, so resolving symlinks against the host filesystem would prove nothing. Decision reasons never repeat a session ID, a path or a message. **Behavior change:** a bare `devsandbox claude` inside a herdr pane previously started no proxy at all; it now starts the filtered proxy, binds the proxy socket into the sandbox, and prints the existing `herdr proxy active` notice. **A herdr control socket that cannot be reached is no longer fatal in the default `auto` mode** - devsandbox warns and continues without the proxy, so a socket left behind by a stopped herdr server cannot break launching a sandbox; `mode = "enforce"`, where the proxy was asked for explicitly, still fails the launch. `devsandbox tools check` reports whether agent session reporting is active and, when it is not, which of the pane or the agent is missing. `claude`, `pi` and `codex` are all recognized, each with the resume shape herdr compiles in (`--resume`, `--session`, and the `resume` **subcommand** respectively).
- **New `devsandbox agent-wrappers install|status|uninstall`, so supported agents run sandboxed by default.** After `install`, typing `claude` runs `devsandbox claude` in the current directory with every argument passed through, while `claude-no-ds` and the shell builtin `command claude` still reach the real binary unsandboxed. Only agents actually installed on the host are wrapped, and devsandbox's absolute path is baked into the snippet so a login shell with a different `PATH` (a mise shim setup, for instance) still resolves it. fish gets a `conf.d` drop-in that activates on the next shell; bash and zsh get a standalone file plus a printed, existence-guarded `source` line to add yourself - **no startup file is ever edited by devsandbox**, and `install`/`uninstall` refuse to touch a file that does not carry the generated header. Nothing is wrapped inside a sandbox: the snippet is guarded on `DEVSANDBOX`, which matters because rc files and fish's `conf.d` are mounted into the sandbox. Scope differs by shell - fish sources `conf.d` for non-interactive `fish -c` too, so fish scripts get the wrapper, while bash and zsh only apply to interactive shells. All three subcommands take `--shell fish|bash|zsh`, defaulting to `$SHELL`. `status` reports the detected shell, the snippet path, whether it is current, which agents are wrapped, which startup file sources it, and - because herdr's native session restore types its resume command into a pane shell - which shell herdr starts panes with, warning when that shell has no snippet installed. The feature is independent of herdr and useful on its own. See [Tools: Shell wrappers](docs/tools.md#shell-wrappers---run-agents-sandboxed-by-default).
- **New `devsandbox run-agent <agent> [args...]` command, the entrypoint for the agent shell wrappers.** It runs a supported agent inside devsandbox from the current directory, forwarding every argument untouched, so `run-agent claude --resume ID` reaches the agent instead of being parsed as a devsandbox flag. Inside a sandbox (`DEVSANDBOX` set to any non-empty value) it executes the real agent directly, so a wrapper that is visible in-sandbox - the shell rc files and fish `conf.d` are bound in - cannot recurse. Re-entry always goes through devsandbox's own executable rather than a `PATH` lookup, so a `devsandbox` earlier in `PATH` cannot redirect it. An agent name devsandbox does not recognize is rejected.
- **Launching an agent from a herdr pane now records which sandbox owns that pane, and `run-agent` refuses a restore that would silently start a new session.** `devsandbox claude` inside a herdr pane writes a small record under `$XDG_STATE_HOME/devsandbox/herdr-panes/` (`~/.local/state/devsandbox/herdr-panes/` when unset; `0700` directory, `0600` files) naming the pane, the agent, the project directory and the sandbox state root the session actually used. When herdr types a resume-shaped invocation (`claude --resume ID`, `codex resume ID`) into that pane, `run-agent` compares the record against the current directory and fails with an actionable error unless re-entry would reach the same project directory *and* the same sandbox state root. A `--worktree` launch is the case that makes this necessary: its state root is derived from the repo root while the session runs in the worktree, so re-entering from either directory opens a different agent session store and would begin a *fresh* session while appearing to resume. A resume typed from an unrelated directory is refused too, with the remedy that actually applies. Every other case re-enters the sandbox as before: a resume with no recorded mapping, a mapping for another agent, a stale mapping, non-resume arguments, or a resume typed outside a herdr pane. A record that exists but cannot be read fails closed rather than falling through. No branch runs an agent on the host, and no error includes the session ID. The pane ID is hashed into the filename, so an opaque herdr-supplied string never influences a path, and the store lives outside every path mounted into the sandbox, so sandboxed code cannot write it. Records are not pruned by `devsandbox sandboxes prune` or `--rm`; delete the directory to clear them.
- **The bwrap backend can now cap memory, CPU and process count.** A runaway build or a fork bomb inside the sandbox previously consumed host resources until the machine became unusable - docker and krun capped memory and CPU, but bwrap, the default backend on Linux, capped nothing. Configure `memory`, `cpus` and `pids` under `[sandbox.resources]` and the sandbox runs inside a systemd transient scope (`systemd-run --user --scope`) carrying the equivalent cgroup v2 controls. `systemd-run` execs in place, so no supervisor process is added and exit codes propagate unchanged; in proxy mode the scope wraps `pasta` rather than devsandbox, so the proxy is not capped by the sandbox's memory limit. **Limits are opt-in with no bwrap defaults** - with no `[sandbox.resources]` block the launch is byte-for-byte what it was, and systemd is not involved at all. **A limit that cannot be enforced aborts the run** rather than silently running unlimited: devsandbox checks for cgroup v2, `systemd-run`, a running systemd user manager and the delegation of each required controller (`memory`/`cpu`/`pids`) to `user@<uid>.service`, rejects values that would round to nothing (`cpus = "0.004"` becoming `CPUQuota=0%`), and creates a throwaway probe scope so a property the user manager refuses at D-Bus time is reported by name instead of arriving later as an ambiguous exit status. Errors name exactly what is missing. Requires cgroup v2 and a systemd user session; hosts without one keep working as long as no limits are configured. **`memory` bounds resident memory on bwrap but does not bound swap** - systemd's `MemoryMax=` leaves `memory.swap.max` at `max`, so on a host with swap a runaway allocator is throttled into swap rather than killed. This is weaker than docker and krun, where `--memory` with `--memory-swap` unset caps memory+swap at twice the value (`memory = "4g"` allows 4g resident plus 4g swap); on a host without swap the backends behave identically. `memory = "0"`, which docker documents as *unlimited*, is **rejected** on the bwrap path, because `MemoryMax=0` would leave the sandbox no memory at all and it would be OOM-killed on start - omit the key for unlimited. The deprecated `[sandbox.docker.resources]` block stays scoped to the container backends and does **not** opt bwrap in: bwrap never honored it, and since an unenforceable limit aborts the launch, applying it there would turn a config written for docker into a failed startup on hosts without a systemd user session. Enabling bwrap enforcement takes an explicit `[sandbox.resources]` block. See [Sandboxing: Resource Limits](docs/sandboxing.md#resource-limits) and [Configuration](docs/configuration.md#resource-limits).
- **The docker backend now honors the `pids` limit.** `pids` under `[sandbox.resources]` becomes `--pids-limit` on the container, capping how many processes and threads the sandbox can create, so a fork bomb inside it can no longer exhaust the host's PID space. It applies to both fresh runs and kept containers, and changing it recreates a kept container instead of reusing one still running under the old limit. Because the value is now part of the container config hash, the first run after upgrading recreates any kept container once, even for users who never set `pids`. **`pids` is not applied on the krun backend, which warns when you configure it** - a krun sandbox is a microVM, so the flag would cap the host-side VMM's own threads rather than the processes inside the guest, whose PID space belongs to the guest kernel and cannot be limited from the host. Rather than break the VM while protecting nothing, krun skips the flag and says so at startup; `memory` and `cpus` are unaffected there. Use bwrap or docker when you need a pids limit.
- **herdr terminal workspace support, through a capability-filtering proxy.** Running devsandbox inside a [herdr](https://herdr.dev) session now lets sandboxed tools open a review overlay in a herdr tab - `revdiff` is the first consumer. The host control socket is **not** bind-mounted; a proxy is, and the sandboxed `herdr` CLI is pointed at it via `HERDR_SOCKET_PATH`. herdr v0.7.4 (protocol 16) exposes 86 methods, including `pane.read` (read any pane), `agent.send`, `worktree.*`, `plugin.*` and `server.stop`; the proxy permits only what enabled tools declare - `tab.create`, `pane.send_input` and `tab.close` for `launch_overlay`, plus `notification.show` for `notify` - and denies everything else. Mutations are scoped to the tab and pane the sandbox itself created, taken from the server's response rather than from anything the client claims, and `pane.send_input` is further restricted to one declared command plus `Enter`. Because herdr runs launch payloads as `sh <path>` rather than inline, the proxy reads the script once, validates it, and copies it to a host-only directory before execution, so the sandbox cannot swap the contents after validation. Configure with `[tools.herdr] mode = "auto" | "disabled" | "enforce"`; see [herdr Terminal Workspace](docs/tools.md#herdr-terminal-workspace). `herdr status` and `herdr session list` report correctly from inside the sandbox: `ping` is permitted unconditionally (a liveness handshake that reveals less than the connect it replaces), and the proxy socket is also exposed at the default `~/.config/herdr/herdr.sock` path that some subcommands resolve on their own instead of honoring `HERDR_SOCKET_PATH`.
- **The container backends (`docker` and `krun`) now make host-installed mise tools available inside the sandbox on Linux hosts.** Previously the guest only had the image's pre-baked node plus whatever the project's `.mise.toml` installed at startup - a network-dependent reinstall of toolchains you already have, and the main reason a mise-centric workflow felt unusable under `krun`. The host's `~/.local/share/mise/installs` is now mounted read-only into the guest, and on startup the in-guest shim mirrors each host-installed version into the persistent sandbox mise data dir as a real version directory whose contents are symlinks into the read-only mount (a fully symlinked version dir breaks mise's aqua/ubi bin-path discovery for nested-layout tools like `uv` and `golangci-lint`; the child-level shape keeps them working), never clobbering sandbox-local installs, and regenerates the mise shims, so your host toolchain (go, node, task, ...) resolves inside the sandbox without reinstalling or network access. The guest also prefers the **host's own mise binary** (mounted via `~/.local/bin`, probe-gated with fallback to the image's) over the image's latest-at-build-time one: a guest mise newer than the host's re-maps a tool's stored backend to its current registry default (e.g. stored `ubi:sharkdp/bat` to registry `aqua:sharkdp/bat`) and then derives bin paths from an archive layout that does not match what is on disk, silently breaking host-installed tools like `bat` or `eza` - version parity removes that skew, and a mise-binary change forces a one-time shim regeneration in existing sandbox homes - this also removes the main trigger for the global-config `@latest` resolution hangs, since the tools those specs resolve to are already present. Versions installed inside the sandbox still take precedence and persist as before. Caveats: host tools compiled locally against a newer glibc than the guest image may fail in the guest (upstream prebuilt tools are unaffected; `mise uninstall <tool>@<version>` + `mise install` inside the sandbox yields a guest-local build - the uninstall only removes the seeded symlink, never the host install), and macOS hosts share nothing (the guest is Linux and cannot run darwin binaries). Only the installs directory itself is shared, so a tool whose runtime lives outside it does not work in-guest: a `pipx:` tool backed by a uv-managed Python, for example, is seeded as a version dir but its interpreter is not mounted, and it fails to start. Reinstall such a tool inside the sandbox. See [Tools: mise](docs/tools.md#tool-management-with-mise).
- **`tools.mise.ignore_global_config` config option to stop the sandbox reading the host's global mise config.** A large host `~/.config/mise/config.toml` with `@latest` `npm:`/`go:`/`pipx:` tools made the sandbox hang and could OOM the guest: every shell start ran `mise activate`, which resolved each `@latest` spec over the network (`npm view …`), and on a proxy/egress-locked sandbox those lookups time out (20s each) while a swarm of them exhausts guest memory. Setting `[tools.mise] ignore_global_config = true` points `MISE_GLOBAL_CONFIG_FILE` at `/dev/null` in the sandbox so the host global `config.toml` tool list is not read; the project `.mise.toml`, the image's system config (baked node), and `~/.config/mise/settings.toml` still apply. It defaults to `false` (the global config is respected, preserving current behavior) and applies to every backend (`bwrap`, `docker`, `krun`). The setting scopes shell and runtime mise invocations inside the sandbox; the docker backend's boot-time project-tool install has always run with `MISE_GLOBAL_CONFIG_FILE=/dev/null` and is unaffected either way.
- **Experimental `krun` microVM isolation backend.** A third isolation backend (`--isolation krun` / `sandbox.isolation = "krun"`) runs the same sandbox image inside a [libkrun](https://github.com/containers/libkrun) microVM via `podman --runtime krun`. Unlike bwrap and Docker, which share the host kernel, krun gives the workload its own guest kernel behind a hardware virtualization boundary (KVM on Linux, Hypervisor.framework on macOS) - the right boundary for running genuinely untrusted code, where a host-kernel exploit must not reach the host. It is opt-in (never auto-selected) and ephemeral (a fresh microVM per launch), runs rootless via `podman --userns=keep-id` so workload-created files stay owned by you, and supports proxy mode on Linux with the MITM proxy bound to host loopback (reached through the pasta gateway, never LAN-exposed). krun + proxy is refused fail-closed on macOS, because the egress lockdown below is Linux-only route surgery with no Hypervisor.framework equivalent yet; non-proxy krun on macOS is unaffected. Requires `podman`, a `crun` built with libkrun (provides the `krun` OCI runtime), and `/dev/kvm` on Linux or Apple Silicon on macOS; `devsandbox` fails fast with installation guidance when a prerequisite is missing. When `[sandbox.resources]` is unset, krun applies sane microVM defaults (`memory = "4g"`, `cpus = "2"`) so the guest is neither starved nor oversized; explicit limits are respected, and the bwrap and docker backends get no defaults.
  - **Egress lockdown:** in proxy mode krun enforces a **host-side, deny-by-default** lockdown - after the microVM boots, `devsandbox` enters the VMM's network namespace (via `nsenter --user --net`) and installs a firewall that drops all egress except loopback, established/related return traffic, and TCP to the proxy gateway on the proxy port, so a destination is reachable only if a rule names it and the LAN (a router UI, a NAS, a LAN DNS resolver), cloud metadata at `169.254.169.254` with its IAM credentials, external DNS exfiltration, and every non-proxy gateway port are closed without being enumerated. The guest is given IPv4 only (pasta is invoked with `-4`) so no IPv6 path exists in the first place, and the default route is deleted alongside as defense in depth. The lockdown runs host-side because under libkrun TSI the guest has no routable interface - its `connect()` calls are executed by the VMM process and obey the VMM netns routing, which an in-guest `ip route del` cannot affect. It prefers `nft` and falls back to `iptables`; if neither is available, or the guest PID cannot be resolved to lock the netns, the microVM is torn down and the launch aborts fail-closed. The surgery is scoped to krun + proxy on Linux and grants no in-guest `NET_ADMIN`. The in-guest shim waits for the host to finish, via a sentinel in the sandbox home, before running the workload, so untrusted code never runs while direct egress is still open - and it waits immediately after reading the host UID/GID, ahead of every mise interaction (host-mise preference probe, `mise trust`, install seeding, `mise reshim`), which touches the guest-writable mise binary and data dir that a prior untrusted run could have planted. Because that home is persistent and guest-writable, the gate accepts only a regular file, written by the host with `O_CREATE|O_EXCL` and checked with `Lstat` (never `Stat`), so a symlink or a stale directory left at the sentinel path is neither followed nor accepted as the go-signal; the host clears the path with `RemoveAll`, verifies it is gone, and aborts before boot if it cannot.
  - **Guest tooling:** an egress-locked krun + proxy guest runs mise **offline** (`MISE_OFFLINE=1`), so `@latest`-style specs resolve instantly from installed and cached data instead of stalling on remote lookups that never traverse the proxy - with host installs seeded that is the full host toolchain. krun runs **no** online boot-time install of your project's tools - it is always ephemeral, and the online `mise install` pass only runs on the persistent-container `docker` paths - so a tool your `.mise.toml` pins that is neither seeded from the host nor already in the sandbox home is not fetched for you; install it in-sandbox with an explicit `MISE_OFFLINE=0 mise install ...`. The startup shim also runs its own mise invocations offline, since they are purely local operations.
  - **Overlay copies:** krun copies overlay/`tmpoverlay` tool directories into the guest on start (it cannot mount kernel overlayfs over virtio-fs), and non-regular entries in the source - a live unix socket, FIFO, or device node such as the `~/.claude/channels/matrix/mux.sock` a Claude Code MCP channel plugin creates - are skipped as the runtime artifacts they are, instead of failing the copy with `open ...: no such device or address` and aborting the whole launch; symlinks, directories, and regular files copy as before. Validated on a `/dev/kvm` host.
  - **Management parity:** `devsandbox doctor` reports krun prerequisites (podman, the `krun` runtime, `/dev/kvm`, a `platform` row when the host OS or CPU architecture cannot run krun at all, and on Linux a system `pasta` binary for rootless podman networking, and `/etc/subuid`+`/etc/subgid` ranges for the `--userns=keep-id` mapping) as informational rows that warn rather than fail, printing each unmet row's remediation under a "How to fix" block below the table; `devsandbox sandboxes list`/`sandboxes prune` cover ephemeral krun sandboxes through on-disk metadata; `devsandbox forward` is best-effort (session registration works, but reaching an in-guest listener through the microVM netns is unvalidated). The in-guest shim no longer emits spurious `groupadd`/`useradd` "already exists" warnings under keep-id. Because krun runs one microVM per project at a time, a second launch while a session is already active now fails fast at startup - before the proxy, docker socket proxy, and image build run - with a copy-pasteable `podman rm -f <container>` remediation, instead of aborting only after the image was rebuilt. See [Isolation Backend](docs/configuration.md#krun-microvm-backend-experimental).

### Fixed

- **Proxy mode no longer leaves the host's own loopback services directly reachable from the sandbox.** pasta forwards ports between the sandbox namespace and the host automatically unless told otherwise (`-T`/`-U` default to `auto`), which binds every host-listening port inside the namespace on loopback. Loopback is the one interface the egress firewall must permit, so a local database, another app's dev API or a model server on the host's `127.0.0.1` stayed reachable from inside the sandbox at `127.0.0.1:<port>` even though the same service was correctly refused at the proxy gateway. Proxy-mode launches on both the bwrap and krun backends now disable that automatic forwarding for every protocol without a configured **outbound** [port forwarding rule](configuration.md#port-forwarding); a configured rule is left exactly as it was, and inbound forwarding is unaffected. Non-proxy launches keep pasta's defaults.
- **Claude and Pi sessions started inside the sandbox are no longer discarded on exit.** Both declare a persistent overlay for their session directory (`~/.claude/projects`, `~/.pi/agent/sessions`) under a parent that is a tmpoverlay, but the overlay is skipped when its host source does not exist - so a user who authenticated the agent on the host and only ever runs it sandboxed, which is exactly what `devsandbox agent-wrappers install` sets up, had every transcript vanish with the sandbox. herdr would then hold a captured session ID resolving to nothing and start a fresh conversation while appearing to resume. devsandbox now creates those directories on the host when the agent's own directory exists, matching what Codex already did; `CLAUDE_CONFIG_DIR` and `PI_CODING_AGENT_DIR` are honored. Nothing is created when the agent directory itself is absent: no tmpoverlay is mounted then, so writes already persist.
- **`claude -c` / `claude --continue` are now covered by the herdr worktree guard.** The guard recognized only herdr's own resume argv, but the shell wrapper intercepts everything you type, and `--continue` reopens the same per-project session store `--resume` does. In a pane whose session was launched with `--worktree`, it therefore re-entered a different sandbox and silently began a new conversation instead of continuing - the exact failure the guard exists to prevent.
- **Agent wrappers say what to do after a devsandbox upgrade instead of failing with command-not-found.** The generated snippet bakes in devsandbox's absolute path, which the documented `mise use -g` install resolves to a version-scoped directory; after an upgrade moves the binary, typing `claude` printed a bare command-not-found and left you guessing. The wrapper now checks the baked path and, when it is gone, prints `devsandbox: no executable at <path> - run 'devsandbox agent-wrappers install' to refresh the wrappers` and exits 127. It fails closed on purpose: it neither falls through to the unwrapped agent nor resolves `devsandbox` through `PATH`, since `PATH` may name a project-local bin directory that sandboxed code can write to, and running that binary on the host would defeat the sandbox exactly when the wrapper is meant to enforce it.
- **A herdr pane record that cannot be parsed now names its file.** The failure refuses every resume in that pane, and deleting the file is the only way out - which the old error gave no way to find.
- **Codex sessions started inside the sandbox are no longer discarded on exit, so `codex resume` can find them.** All of `~/.codex` was mounted as a config directory, which the default `split` mount mode gives a tmpoverlay - every rollout file Codex wrote under `~/.codex/sessions` vanished with the sandbox, and a later `codex resume <id>` reported `no rollout found for thread id`. The Codex home keeps its tmpoverlay so `config.toml` and `auth.json` stay shielded, and `~/.codex/sessions` is now bound separately with a persistent overlay (verified against codex-cli 0.144.6: resolving a session walks `sessions/YYYY/MM/DD/` for the rollout file and reads nothing else under the Codex home, so nothing further needs to persist). `CODEX_HOME` is honored for both. devsandbox creates `sessions/` on the host when a Codex home exists but the directory does not, because an overlay whose source is missing is skipped - without it a user who authenticated codex on the host but only ever runs it sandboxed would keep losing every rollout. Nothing is created when the Codex home itself is absent: no tmpoverlay is mounted then, so writes already persist in the sandbox home. `devsandbox tools check codex` lists the sessions directory when it exists.
- **`revdiff` can now actually open its review overlay in a herdr tab.** Every launch was denied with `pane.send_input: command does not match any declared launch pattern`, so under herdr the review never appeared - the launcher only reported `herdr pane run failed`, since it discards the proxy's reason. The launcher shell-quotes the generated script path, sending `sh '<path>'`, but the proxy recognized only the unquoted `sh <path>` form; the quoted text fell through to inline-argv matching, which `revdiff` declares nothing for, and was refused. The proxy now strips one layer of single quotes before validating the path. Quoting is unwrapped only when the result is a plain path - a remainder still containing a quote (`'\''`-style escaping, nested or unbalanced quotes) is left alone and continues to be rejected by the absolute-path and metacharacter checks, so nothing the shell would act on reaches the host. Double-quoted paths remain unsupported. This affected the herdr backend only; the kitty overlay path passes an argv array and never had the problem.
- **The docker backend no longer aborts startup with `fork overlay child: operation not permitted`.** Any sandbox with a `tmpoverlay` tool directory - which the default `split` mount mode gives every config path, so effectively all of them - failed to start on the docker backend. The in-guest shim realized those directories by forking a child into a new `CLONE_NEWUSER|CLONE_NEWNS`, but Docker's default seccomp profile denies `clone`/`unshare` carrying namespace flags unless the container holds `CAP_SYS_ADMIN`, and the sandbox deliberately runs with `--cap-drop ALL` plus only CHOWN/SETUID/SETGID/DAC_OVERRIDE. The container therefore exited during setup, and the only workaround was another backend. `tmpoverlay` directories now use the same copy-on-start path that macOS and krun already used, so no capability or seccomp relaxation is needed. The behavioral difference is where discarded writes live: instead of vanishing with a tmpfs upper layer at exit, they sit in the sandbox home and are cleared at the start of the next run, so a previous run's writes are still never visible to the sandbox. `bwrap` is unaffected and keeps using kernel overlayfs. Note that `auto` selects bwrap on Linux, so this only ever affected sandboxes that opted into `isolation = "docker"` (commonly those needing a custom Dockerfile).
- **The `krun: system pasta` doctor row says when its warning may be a false positive.** The probe only searches `$PATH`, while podman also looks in `helper_binaries_dir` from `containers.conf` (`/usr/libexec/podman` and similar), so a host that installs pasta there was told to install `passt` it already has. The row still warns - it is advisory - but its remediation now names the limitation and gives the command that settles it (`podman info --format '{{.Host.Pasta.Executable}}'`), matching how the `rootless id mapping` row already handles its libsubid/NSS blind spot.
- **The kitty proxy no longer accepts a launch command that names a sandbox-planted binary.** Launch patterns matched `argv[0]` on basename alone, so any path ending in the allowed program name was accepted - including one inside a directory the sandbox can write. That matters because the revdiff IPC directory is a write-through bind mounted at an *identical path* on the host and inside the sandbox (unlike the rest of the sandbox home, whose overlay writes never reach the host). Sandboxed code could therefore drop its own executable at `<ipc-dir>/revdiff`, name it in a `kitty @ launch` request, pass validation on basename, and have kitty run it **on the host as the host user** - a full sandbox escape. Launch patterns are now pinned to the program's resolved absolute path (`exec.LookPath` + symlink evaluation), and patterns may additionally declare directory prefixes that can never supply the program. If the binary cannot be resolved, every launch request is denied rather than falling back to the old basename behavior.
- **A second session for the same project no longer breaks a running session's notifications, Docker access, and kitty remote control.** The portal, Docker, and kitty proxies each created their unix socket at a path keyed only on the project (`<sandbox home>/.dbus-proxy/bus`, `<sandbox home>/docker.sock`, `<sandbox home>/.kitty.sock`), but sandbox home is shared by every session for that project. Starting a second session unlinked the live session's socket and re-created it, and that session's exit deleted the path outright - leaving the first session with `DBUS_SESSION_BUS_ADDRESS`, `DOCKER_HOST`, and `KITTY_LISTEN_ON` pointing at a path that no longer existed. `notify-send` failed with `Could not connect: No such file or directory` for the rest of the session, with no way to recover short of restarting it. Each session's sockets now live in a directory private to the owning process (`<sandbox home>/.run/<pid>/`), so concurrent sessions cannot disturb each other; directories left by sessions that are gone are reclaimed on the next start. `DBUS_SESSION_BUS_ADDRESS` is unchanged; `DOCKER_HOST` and `KITTY_LISTEN_ON` now point at `$HOME/.run/<pid>/docker.sock` and `$HOME/.run/<pid>/kitty.sock` inside the sandbox.
- **A socket path too long for the kernel is now reported as such.** The proxy socket paths above sit under an already-long sandbox home, and `bind(2)` rejects anything past 107 bytes (103 on macOS) with a bare `invalid argument` - which the portal surfaced only as an opaque "proxy socket not created" timeout. The portal, Docker, kitty, and herdr proxies now fail with the path, its length, the limit, and the remedy (a shorter project directory name).
- **Proxy-mode sandboxes no longer stall for minutes resolving `@latest` mise tool specs.** mise refreshes the remote version list behind every `@latest`-style spec in a mise config (host global config included), some backends' lookups (npm registry, python-build) never traverse the proxy and hang to their 20s default timeout, and mise re-resolves the toolset per listed row with no negative cache - measured on an egress-locked guest, a single `mise ls` against a global config with two `@latest` npm specs ran for **14 minutes** across ~300 doomed lookups. All proxy-mode sandboxes now bound remote lookups with `MISE_FETCH_REMOTE_VERSIONS_TIMEOUT=3s` (was 20s), and a value you set through the sandbox env config takes precedence over that default on every backend, as the docs already promised. Under krun the offline-mise layer described above removes the lookups entirely: verified end-to-end on a KVM host, `mise ls` in an egress-locked guest went from 14 minutes with a wall of warnings to under a second with none.
- `tmpoverlay` config dirs that degrade to a copy-on-start overlay (krun on any OS, and the Docker backend on macOS) are now reset to the host source on every run; files written by a previous run no longer persist. On these backends a `tmpoverlay` tool dir cannot use kernel overlayfs (the libkrun guest rejects overlayfs over virtio-fs; Docker Desktop on macOS has no overlayfs), so it degrades to a copy into the *persistent* sandbox home. The copy only wrote source entries and never removed extraneous ones, so anything a previous - possibly untrusted - run left under the target (for example a hook planted in `~/.claude`) survived into the next session, defeating tmpoverlay's discard-on-exit promise. The shim now clears the target before copying; the clear is mount-aware, preserving any nested read-only bindings (e.g. `~/.claude/projects`) and their ancestor directories while removing everything else. Copyoverlay targets are nested (e.g. `~/.config/Claude`, with nothing mounted at `~/.config`), so if a previous run replaced the target - or any intermediate component of its path - with a symlink (pointing at the project dir or another read-write mount), the shim now walks every component from the sandbox home down to the target, removes any symlink it finds - never following it - and recreates a real directory before cleaning and copying, so the root-privileged clear and copy never delete or write outside the target.
- **Docker isolation no longer hangs at startup for non-root users when a tool uses an overlay mount.** The container drops all capabilities and the entrypoint shim runs as container-root to populate `/home/sandboxuser` and then drop privileges, but `CAP_DAC_OVERRIDE` was granted only to the krun backend. Under rootful Docker the home is bind-mounted from the host (owned by your non-root host UID) and the overlay manifest was a host temp file (mode 0600, host-owned), so without `DAC_OVERRIDE` container-root could not create files in the home (silent `fish_variables` / `ssh/environment` warnings) nor read the manifest - the manifest read is fatal, so the shim exited before signalling ready and the launch hung until the 90s readiness timeout. This only surfaced when a tool binding introduced a `tmpoverlay` (which generates the manifest) and went unnoticed because Linux defaults to the bwrap backend. `DAC_OVERRIDE` is now granted on the docker backend too, scoped to the shim's root setup phase: the shim still `setuid`s to the unprivileged user (clearing its capabilities) before exec'ing the workload, with `no-new-privileges` set, so the workload never holds the capability.
- **A kept Docker container is now actually reused when a tool uses an overlay mount, instead of being destroyed and rebuilt on every launch.** The overlay manifest was written to a fresh temp file per run (`/tmp/devsandbox-overlay-manifest-<random>.json`) and deleted on exit, but the container binds that host path permanently. Every later `docker start` therefore mounted a path that no longer existed: Docker materialized the missing bind source as a directory, refused to mount it over a file, and the launch fell back to removing the container and recreating it - `keep_container` (on by default) bought nothing, all container state was lost each run, and the startup printed an alarming `not a directory: Are you trying to mount a directory onto a file` warning. The manifest now lives at a stable per-project path (`~/.local/share/devsandbox/<project>/overlays.json`), is rewritten in place on each launch, and is no longer removed on exit. It is written outside the sandbox home, so the sandbox cannot rewrite it between the host writing it and the shim reading it, and it is now mode 0644 rather than inheriting 0600 from the temp file.
- **A container whose startup fails now reports the failure immediately, instead of after the 90s readiness timeout.** The readiness probe only polled for the shim's ready sentinel via `docker exec`, which fails identically for a container that is still booting and one that has already died - so a shim that fatally errored a second into setup still cost the full 90s wait, and the error said "setup timed out" when nothing had timed out. The probe now also checks container state and fails as soon as the container is seen exited, reporting its exit code alongside the container log tail that already accompanied the timeout. An unreadable container state is not treated as an exit, so a transient inspect failure still falls through to the timeout rather than aborting a healthy boot.
- **Piped stdin now reaches non-interactive krun (and docker) commands.** `devsandbox --isolation krun - cmd` ran the container without `-i`, so `data | devsandbox --isolation krun - tool` closed the workload's stdin and the piped input was silently lost (the bwrap backend, which runs the workload as a direct child, was unaffected). The container/exec is now started with `-i` for non-interactive runs, so stdin is forwarded on every backend. A TTY is added on the run and exec paths only for interactive sessions; the default `keep_container` create path still allocates one unconditionally (`-it`), because a container created for a one-off command may later be reused for an interactive shell.
- **A sandboxed command's exit code now propagates to the host instead of collapsing to `1`.** `devsandbox - sh -c 'exit 42'` exited `1` on every backend (only `0` passed through), because a non-zero command result was treated as a generic CLI error. The workload's exit status is now carried up and re-emitted as `devsandbox`'s own exit code (so `42` stays `42`), and a non-zero command exit no longer prints a spurious `Error:` line - matching how a shell surfaces a child's status. This now covers the **default `keep_container` create/exec path** on the docker backend too (`podman exec` was returned unwrapped, so its status still collapsed to `1` there). Genuine devsandbox setup failures (image build, container create, egress lockdown) still surface loudly and exit `1`; a container-engine launch failure (`podman`/`docker` exit `125` - a bad flag or a create/exec that never started the workload) is likewise surfaced as an error rather than propagated silently as if it were the command's own status.
- **The session/proxy lock files are no longer unlinked on release, and are opened with `O_NOFOLLOW`.** The file lock (used by the proxy lifecycle lock and the new krun per-project session lock) unlinked its lock file on release, which reopened a split-lock race: a second process could `flock` the old inode after the holder unlocked but before it unlinked, the unlink then removed a path a third process recreated and flocked independently, and two holders ran at once. The lock file is now created once and left in place - the kernel auto-releases the `flock` when the holder exits, so there is no stale file to clean up - and it is opened with `O_NOFOLLOW` so a co-tenant cannot pre-plant a symlink at the predictable temp path to make the (possibly privileged) holder truncate an arbitrary file.
- **The in-guest shim no longer silently discards sandbox user/group creation errors, and `USER` names the real account.** A failed `useradd`/`groupadd` (e.g. the uid or the `sandboxuser` name already claimed under a different name) was ignored, and `USER` was hardcoded to `sandboxuser` even when no such passwd entry existed - so `USER` could name an account that was not there. Creation failures are now warned (never silent), and `USER` is resolved from the passwd entry for the uid the shim drops to, falling back to `sandboxuser` only when there is genuinely no entry.
- **The container backends no longer reinstall the pre-baked node on every guest, and the new krun backend persists mise-installed tools across runs like docker already did.** `MISE_DATA_DIR` points at the sandbox home (`~/.local/share/mise`), which is bind-mounted and persistent, so a tool the project installs inside the sandbox (python, go, etc.) is installed once and reused on later runs instead of being re-downloaded and re-extracted on every fresh guest - unchanged for docker, and now the same under krun. What is new on both is the pre-baked node: the image's `node@22` (which the AI CLIs run on) lives in the ephemeral image path (`/opt/mise`), so to keep it resolving instantly the in-guest shim now mirrors it into the persistent data dir as version-level symlinks on startup; without this the baked node is orphaned and every fresh guest stalls for minutes reinstalling it - badly so alongside a large global `~/.config/mise` config, where the reinstall also dragged in unresolvable `@latest` `npm:`/`go:`/`pipx:` tools and produced `node is not a valid shim` / `Failed to resolve tool version` errors. The seed is version-level (not whole-tool) so a node version the project installs itself still persists to the home rather than being redirected back into the ephemeral image. Your global mise config is still respected; the download cache is still persisted across runs; the bwrap backend, which uses the host's real mise install, is unchanged.
- **krun now refuses to launch on an Intel Mac instead of failing obscurely later.** libkrun's macOS backend is Hypervisor.framework on Apple Silicon, and there is no supported path on `darwin/amd64`, but the prerequisite check had no architecture probe - so an Intel Mac passed preflight and the launch only broke after the image build, with an opaque runtime error. The krun prerequisites now include the host CPU architecture, checked first so the failure names the unusable hardware rather than a tool you would install for nothing, and the message points at `--isolation=docker` on Intel Macs or a Linux host with `/dev/kvm`. `devsandbox doctor` reports the same as a `krun: platform` row.
- **`devsandbox config init` no longer emits obsolete config keys.** The generated template documented an `[overlay] enabled` switch that no longer exists (the key is `default`, naming the mount-mode policy) and emitted `[tools.mise] writable` / `persistent`, neither of which mise reads any more - so copying the template's own suggestions produced settings that silently did nothing. The template now documents `overlay.default` with its five modes and the real mise keys (`mount_mode`, `ignore_global_config`), and a round-trip test asserts every key the generator emits is one the loader recognizes.
- **A `hidden` mount rule that hides nothing is now reported at launch, and its remediation now names something that actually works.** A rule whose pattern resolves to a *directory* cannot be applied - a directory cannot be replaced by `/dev/null` - and the skip was only written to the sandbox log file, so a user who wrote `pattern = "secrets/**"`, `mode = "hidden"` got a fully readable secrets directory with nothing on the terminal to say so. The skip is now a startup warning on stderr. Its old advice was wrong on top of being invisible: it named `readonly`, `overlay`, and `tmpoverlay`, none of which conceal a directory's contents (they all keep the host files readable). The warning, `docs/configuration.md`, `docs/sandboxing.md`, and the config schema now give the only remedy there is - match the *files inside* the directory (`**/secrets/**`, `~/secrets/**/*`), which is what the working examples already did. Directories whose files the same rule hid are not warned about, so the documented `**/secrets/**` pattern stays quiet.
- **`devsandbox doctor` no longer fails the run because past sandbox runs logged errors.** The `logs` row escalated to an error above 10 errors in the last 24h across all projects, so `doctor` exited `1` with "Please install missing dependencies" and no hint - on a host where every other row was ok and nothing was missing. Recent log errors say nothing about whether a sandbox can launch now, so the row is advisory (`warn`) at any count, matching the `docker`/`docker-image` rows, and carries a hint pointing at `devsandbox logs internal --since 24h`. This makes `doctor` usable as a CI/script gate again. The failure summary also changed: it names the rows that failed and only advises installing something when a row actually reports a missing dependency (`bwrap`, the shell), instead of blaming missing dependencies for an unwritable directory or an unreadable config.
- **The `hidden` mount mode is now documented as files-only, and the README no longer claims the sandbox exposes "nothing else".** The `devsandbox config init` template and the `MountRule.Mode` schema comment both described `hidden` as hiding "the file/directory", but the builder skips directory matches and only overlays files with `/dev/null` - so a rule aimed at a secrets *directory* left it fully readable inside the sandbox, and the template's own example advertised exactly that. Both surfaces now state files-only, and every surface now names the remedy that works (see the hidden-directory entry above). Separately, the README's isolation summary said the agent sees the project directory and your tools "and nothing else", while the defaults also mount read-only host system paths (`/usr`, `/lib`, `/bin`, a few `/etc` files) and the shell, editor, and sanitized git configs those tools need; it now names those and links the [mount layout](docs/sandboxing.md#filesystem-isolation). What is *not* mounted - SSH keys, cloud credentials, sibling projects, parent directories - is unchanged. Documentation only - no behavior changed.

### Changed

- **bwrap proxy mode is now enforced, not best-effort.** Proxy mode on the default backend used to apply route surgery that discarded its own errors and exec'd the workload regardless, so a namespace where the surgery did not apply - including any host where `ip` lives in `/usr/sbin` and is not on the user's `PATH` - started with **egress fully open** and nothing reported it. Even where it applied, hosts on the sandbox's own connected subnet stayed reachable (deleting the default route does not remove the on-link route, so the router UI, a NAS, the LAN DNS resolver, and a cloud host's `169.254.169.254` metadata endpoint were all still directly reachable), every port of the host's loopback was reachable at the gateway address rather than only the proxy port, and IPv6 egress was untouched. A proxy-mode sandbox now gets the same deny-by-default lockdown krun already applies: only loopback, established/related return traffic, and TCP to the proxy port on the gateway are permitted; everything else has no direct path out. This scopes the sandbox's **path** to the proxy, not its reachable **destinations**: what the proxy will connect to on the sandbox's behalf is still decided by [HTTP filtering](docs/proxy.md#http-filtering), which allows everything until `default_action` is set - so a metadata or LAN address is refused as a direct socket and still served through the proxy, where it is at least visible and refusable. The rules are applied inside the network namespace **before the sandboxed workload exists**, so there is no window in which untrusted code runs with egress open, and any failing step aborts the launch with a `devsandbox: egress lockdown:` diagnostic instead of continuing. Direct DNS is deliberately not excepted - the proxy resolves hostnames itself, and permitting `:53` would re-open a DNS-tunnel exfiltration channel. Configured **outbound** `[[port_forwarding.rules]]` keep working: each one adds an accept for exactly its port and protocol on the gateway. Host loopback ports that were *not* configured as an outbound rule are no longer reachable at `10.0.2.2` - closing that exposure is the point of the change, and declaring the rule is the fix. The rule set is shared with krun, so the two backends cannot drift apart. See [Backend-specific behavior](docs/proxy.md#backend-specific-behavior).
- **`devsandbox doctor` checks the proxy firewall prerequisite for every backend, and checks it by using it.** The old `krun: firewall` row only looked for an `nft` or `iptables` binary, and its name told bwrap users - the default backend, which now also hard-aborts a `--proxy` launch without a working firewall - that it did not concern them. It is now a top-level `proxy: firewall` row that applies the actual lockdown rule set in a throwaway network namespace, so a host that has `nftables` installed but cannot load `nf_conntrack` is reported here instead of failing mid-launch. The row stays advisory (a warning, never a failure): proxy mode is opt-in, and launches without `--proxy` need none of this. Its remediation names both halves - install `nftables`/`iptables`, and `sudo modprobe nf_tables nf_conntrack`.
- **Resource limits moved to a backend-neutral `[sandbox.resources]` section, and gained a `pids` limit.** Limits used to live under `[sandbox.docker.resources]`, which named the docker backend even though krun read the same values and bwrap could not use them at all. The new section carries `memory`, `cpus` and the new `pids` (maximum processes/threads) and is honored by every isolation backend. `[sandbox.docker.resources]` still works - it is **deprecated**, and it stays scoped to the container backends: `docker` and `krun` merge it field by field with the new section, with the new section winning on any field it sets, so on those backends adding `pids = 2048` under `[sandbox.resources]` does not discard a `memory` value you already have under the old block. **`bwrap` does not read the deprecated block at all**, so on bwrap a `[sandbox.resources]` block is the whole configuration - a `memory` value left behind under `[sandbox.docker.resources]` is ignored there and must be moved. That carve-out is deliberate: bwrap never honored the docker-scoped section, and since it aborts the launch when a limit cannot be enforced, applying the old section to it would turn a working docker-oriented config into a failed bwrap startup on any host without a systemd user session. Using the deprecated block prints one warning at startup, regardless of how many config files set it. Validation errors now name the TOML block they came from (e.g. `invalid [sandbox.resources] cpu limit`) instead of saying "docker" for both. See [Resource Limits](docs/configuration.md#resource-limits).
- **`devsandbox doctor` no longer closes with "All checks passed!" when rows warned.** A run with no failures but one or more warnings now ends with `All required checks passed (N advisory warning(s))`, extended with `- see "How to fix" above` when any of those rows carries remediation. Most hosts see this: the krun rows (podman, the `krun` runtime, system `pasta`, subordinate id ranges) and the not-pulled-locally `docker-image` row warn on a machine that never uses those backends, and previously the run claimed everything passed directly below a "How to fix" block listing what did not. The exit code is unchanged - only `error` rows fail the run, warnings stay advisory - and a genuinely clean run still prints "All checks passed!". The "How to fix" block also moved to sit directly above that closing line, instead of being separated from it by the "Detected Tools" table. A failing run whose remediation entries are all advisory (e.g. `bwrap` missing on a host that also warns about the opt-in krun rows) now heads the block "How to fix (advisory warnings):" and says so in the closing line, so the krun advice is not read as the fix for what failed; when a failing row carries its own remediation the closing line points at the block instead.
- Embedded `pasta` upgraded to passt `2026_07_16.090d739` (from `2026_06_11.a9c61ff`). The statically linked pasta binary that backs sandbox networking is rebuilt from the newer upstream passt release. Embedded `bwrap` is unchanged at `v0.11.2`.
- **krun warns when building a project-provided Dockerfile, whose build runs outside the microVM boundary.** A custom Dockerfile (the `sandbox.docker.dockerfile` config key) is built by host `podman build` before the microVM boots, so its `RUN` steps execute on the host - outside the krun guest isolation and outside the proxy egress lockdown. krun now prints a warning before such a build, advising that only trusted Dockerfiles be used; the auto-generated default (config-dir) Dockerfile and every docker-backend build stay silent. This is a trust-boundary disclosure, not a behavior change - the build still proceeds. See [Build-time trust boundary](docs/getting-started/krun.md#build-time-trust-boundary).
- **Docker/krun startup timeouts now report the container's logs instead of a bare deadline.** On the Docker (and krun) backends the container runs detached while `devsandbox` polls for a readiness sentinel, so when the in-guest shim fails before signalling ready (e.g. `read overlay manifest: permission denied`), its fatal output went to the container log and never reached the user - the launch only surfaced an opaque `container setup timed out after 90s`. The timeout error now appends the last 50 lines of `docker logs`/`podman logs` for the stuck container, so the actual cause is visible without re-running the engine by hand. The krun port-forward PID-resolution timeout includes the same log tail in its warning. Log retrieval is best-effort and never masks the original timeout.
- **Proxy-mode, content-redaction, and `.env` masking claims now state their actual limits wherever they appear.** `devsandbox --help` and `--info`, the `devsandbox config init` template, the README, the landing page, the quickstart and first-run guides, `docs/sandboxing.md`'s security model table, and `docs/proxy.md`'s bwrap walkthrough each described proxy mode as routing or logging *all* HTTP(S) traffic and `.env` masking as unconditional. Neither holds: Docker is env-var routing with no network-level enforcement, and only bwrap and krun fail closed (bwrap's route surgery was best-effort until the egress lockdown above landed in this same release); `.env` masking scans 3 directory levels below the project root and skips `node_modules`, `.git`, `vendor` and `.venv`. Content redaction was likewise described as scanning outgoing requests "before they leave your machine" with no scope: it only sees requests that reach the proxy - so it inherits the same per-backend gaps - and HTTPS bodies, headers, and URLs only when MITM is enabled. `docs/proxy.md` gained a [Redaction Coverage](docs/proxy.md#redaction-coverage) section with the full detail, and the README, `docs/configuration.md`, and the config template now point at it. The bwrap proxy-mode requirement also no longer claims a `passt` system package is needed - pasta is embedded and extracted on first use, and the package is only a fallback when `use_embedded = false` or extraction fails. These surfaces now carry the qualifier and point at [per-backend behavior](docs/proxy.md#backend-specific-behavior). Documentation only - no behavior changed.

## [v0.17.3](https://github.com/zekker6/devsandbox/releases/tag/v0.17.3) - 2026-06-24

### Added

- **`DEVSANDBOX_DEBUG=1` proxy lifecycle tracing.** The MITM proxy now logs a per-request `CONNECT` / `request` / `response` trace to the internal proxy log (`devsandbox logs internal --type proxy`), including response status, content-type, streaming detection, and time-to-headers. Query strings are stripped so tokens are never logged. Use it to pinpoint where a hung or timed-out request stalls. See [Debugging the Request/Response Lifecycle](docs/proxy.md#debugging-the-requestresponse-lifecycle).

### Fixed

- **MITM proxy no longer buffers response bodies before relaying headers.** goproxy relays a response to the client only after the `OnResponse` handler returns and does not flush the body until the handler-supplied `resp.Body` is read, but request logging read the entire body with `io.ReadAll` to capture it. For any streaming response the body stays open until generation finishes, so the proxy withheld the response *headers* for the full duration - codex aborted with `Codex SSE response headers timed out after 20000ms` while the proxy spent 10-80s reading the stream (one HTTP upgrade buffered for 82s). Crucially these responses are not always identifiable by `Content-Type` (codex's streamed responses carry an empty `Content-Type`), so media-type sniffing alone could not avoid the buffering. The response body is now wrapped so it streams to the client unchanged while a bounded prefix (256 KiB) is captured for logging; the log entry is written when the body closes. The proxy never buffers a body before relaying headers, so SSE, chunked, empty-`Content-Type`, and large responses all stream incrementally. This also unbreaks **WebSocket (WSS) and other HTTP upgrades through MITM**: 1xx/`101 Switching Protocols` responses are now left untouched, so goproxy can type-assert `resp.Body` to `io.ReadWriter` and relay the upgraded connection (the old code read the 101 body and replaced it with a `bytes.Reader`, stalling the stream for up to 82s and failing the relay).

## [v0.17.1](https://github.com/zekker6/devsandbox/releases/tag/v0.17.1) - 2026-05-13

### Fixed

- **Proxy no longer panics on requests with a nil `URL`.** goproxy can dispatch HTTPS requests whose `http.Request.URL` is nil when its internal `url.Parse` fallback fails (the parse error is swallowed and the request is still handed off). Every downstream step - credential injection, filtering, redaction, ask-mode, request logging - dereferences `req.URL`, so any such request crashed the proxy worker. `RequestLogger.LogRequest` now falls back to `RequestURI` when `URL` is nil, and the request handler short-circuits with a 403 (`malformed request: missing URL`) instead of dispatching downstream.

## [v0.13.1](https://github.com/zekker6/devsandbox/releases/tag/v0.13.1) - 2026-04-18

### Fixed

- `revdiff` tool no longer wipes its shared IPC directory on `Start`/`Stop`. Because the dir is exported as `$TMPDIR` for every sandboxed process, long-lived tenants (Claude Code's per-session task cache under `$TMPDIR/claude-<uid>/…/tasks/`, Node's compile cache, Go's build cache) populate subtrees that must survive sandbox restarts for the same project - and parallel sandboxes on the same project share the directory, so wiping it from one tore state out from under the others. The old `RemoveAll` on `Start` could yank state out from under a running caller; Node's non-recursive `fs.mkdirSync` then failed with `ENOENT`, breaking every subsequent Claude Code Bash tool call. `Start` now only ensures the dir exists (0700); `Stop` is a no-op. Stale revdiff sentinels are harmless - the launcher uses `mktemp` with fresh names.

## [v0.13.0](https://github.com/zekker6/devsandbox/releases/tag/v0.13.0) - 2026-04-17

### Added

- `[sandbox.environment.<NAME>]` config block: declare sandbox environment variables using the same source model as proxy credentials (`value` / `env` / `file`, priority `value > env > file`). `env = "X"` with `X` unset on the host silently skips the variable; an unreadable `file = "..."` is a startup error. Declaring the same variable in both `env_passthrough` and `environment` fails at startup with a message naming the variable - each variable belongs in exactly one place.
- `pi` tool: integrates [Pi Coding Agent](https://github.com/badlogic/pi-mono). `~/.pi/agent` is mounted with credential protection; `~/.pi/agent/sessions` persists across runs.
- `[proxy.credentials.github] overwrite = true`: force-replace any existing `Authorization` header on outgoing `api.github.com` requests. Intended for the pattern where a sandboxed CLI (e.g. `gh`) refuses to start without a token in its environment - pass a placeholder through `env_passthrough` / `sandbox.environment` while the real token stays on the host and is swapped in by the proxy. Default remains `false` (existing tool-set headers are preserved).
- `revdiff` tool now provides a shared IPC directory (`~/.cache/devsandbox/revdiff-ipc/<session>/`) bind-mounted at the same path on both sides and exported as `TMPDIR`. The kitty-spawned overlay shell runs on the host and receives sentinel/output paths as literal strings, so host and sandbox must agree on the string - argv-shipped paths need `Source == Dest` equality, not just a shared inode.

### Changed

- **Kitty tool now runs a capability-filtering proxy instead of bind-mounting the host socket.** The host kitty remote-control socket is no longer exposed inside the sandbox; a local proxy at `$HOME/.kitty.sock` is exposed instead, and `KITTY_LISTEN_ON` is rewritten to point at it. Sandboxed processes can only issue kitty commands declared as capabilities by an enabled tool (`launch_overlay`, `launch_window`, `launch_tab`, `launch_os_window`, `close_owned`, `wait_owned`, `focus_owned`, `send_text_owned`, `get_text_owned`, `set_title_owned`, `list_owned`), and `*_owned` commands are scoped by ownership tracking to windows the sandbox itself opened. Shell metacharacters in `sh -c` payloads for `launch_*` are rejected outright. `remote_control_password` is unsupported - use `allow_remote_control = socket-only`. New `[tools.kitty]` fields: `mode` (`auto` default / `disabled` / `enforce`) and `extra_capabilities` (additive; `launch_*` entries rejected). Under `auto`, the proxy only starts when at least one enabled tool declares a capability - zero attack surface when no tool needs kitty. `revdiff` is the built-in consumer.

### Fixed

- macOS: shortened test directory names to stay under the platform's unix socket path length limit (affected `kittyproxy` and `kitty` tool tests).

## [v0.12.0](https://github.com/zekker6/devsandbox/releases/tag/v0.12.0) - 2026-04-16

### Changed

- Wrapper diagnostic output (port-forward notices, session warnings, proxy setup info, container progress) no longer writes directly to stderr while a child process owns the terminal. Messages are written to `$XDG_STATE_HOME/devsandbox/wrapper.log` (or `~/.local/state/devsandbox/wrapper.log`) and a one-line banner is shown on exit if anything was suppressed. This prevents wrapper output from corrupting TUI applications (Claude Code, aider, etc.) running inside the sandbox. Pass `--verbose` or set `DEVSANDBOX_DEBUG=1` to restore the old behavior of writing every message to stderr.

### Added

- `--worktree` and `--worktree-base` flags: opt-in git-worktree mode. Bare `--worktree` auto-generates `devsandbox/<session-or-timestamp>` off HEAD; `--worktree=<branch>` reuses or creates a named branch. The sandbox CWD is the worktree; the main checkout is untouched. With `--git-mode=readwrite`, commits land on the worktree branch only. `--rm` removes the worktree on exit via `git worktree remove --force` + `prune`. `--worktree` + `--git-mode=disabled` is rejected at flag-parse time. Worktrees live at `~/.local/share/devsandbox/<project-slug>/worktrees/<branch>/` and the slug is derived from the main repo root so sibling worktrees share sandbox state. `devsandbox sandboxes prune` and the `doctor` command are worktree-aware.

### Fixed

- `zellij` and `kitty` tool socket bindings are now explicit bind mounts (`Type: MountBind`) instead of inheriting the default tmpoverlay from `CategoryRuntime`. Overlayfs cannot expose a unix socket from its lower layer, so under the previous policy the host socket was invisible inside the sandbox and `zellij list-sessions` / `kitten @` silently failed.
- Auto port-forwarding no longer tries (and fails) to forward when the sandbox shares the host network namespace. Without proxy mode the sandbox uses bwrap's `--share-net`, so a tool listener inside the sandbox is the same kernel socket as the "host" bind the forwarder would attempt - producing a spurious `bind: address already in use` error for every detected port. Auto-detect now inspects the sandbox netns inode and skips forwarding (with a one-line explanatory message) when it matches the host; the sandbox ports are already directly reachable on `127.0.0.1`. For the rare case where auto-forward runs in a properly isolated netns but the host happens to already have that port in use, the forwarder falls back to an ephemeral host port and logs the mapping instead of silently dropping the service.

## [v0.8.0](https://github.com/zekker6/devsandbox/releases/tag/v0.8.0) - 2026-04-05

### Breaking Changes

- **`[overlay] enabled` removed** - replaced by `[overlay] default` which accepts:
  `split` (default), `overlay`, `tmpoverlay`, `readonly`, `readwrite`.
- **`[tools.mise] writable` and `persistent` removed** - use `[tools.mise] mount_mode` instead.
  Mise no longer has tool-specific overlay configuration; use the unified `mount_mode` system.
- **Default mount behavior changed** - tool mounts now default to `split` overlay policy
  (configs → tmpoverlay, caches/data/state → persistent overlay) instead of read-only
  bind mounts. This prevents supply chain attacks from poisoning host tool configurations
  through sandboxed package managers.

### Migration Guide

| Before | After |
|--------|-------|
| `[overlay] enabled = true` | `[overlay] default = "split"` (or omit - it's the default) |
| `[overlay] enabled = false` | `[overlay] default = "readonly"` |
| `[tools.mise] writable = true, persistent = true` | `[tools.mise] mount_mode = "overlay"` |
| `[tools.mise] writable = true, persistent = false` | `[tools.mise] mount_mode = "tmpoverlay"` |
| (no equivalent) | `[tools.git] mount_mode = "readwrite"` |

### Added

- **Binding categories** - tools now classify each mount as `config`, `cache`, `data`,
  `state`, or `runtime`, enabling differentiated overlay policies.
- **`[overlay] default`** - global mount mode for all tool bindings with five modes:
  `split`, `overlay`, `tmpoverlay`, `readonly`, `readwrite`.
- **Per-tool `mount_mode`** - override the global default for specific tools
  (e.g., `[tools.git] mount_mode = "readwrite"`). Accepts `disabled` to prevent
  a tool's config from being mounted entirely.

### Changed

- Tool bindings no longer hardcode `ReadOnly` or `Type` - the builder resolves these
  based on the mount mode policy chain (per-tool > global > split).
- Claude, Copilot, Codex, and OpenCode tools previously mounted configs read-write;
  they now follow the global mount mode (default: tmpoverlay for configs).
